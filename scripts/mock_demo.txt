# Sample mock provider script. Drives hax through a sequence of canned
# turns that exercise the rendering pipeline end-to-end without ever
# talking to a real model.
#
#   HAX_PROVIDER=mock HAX_MOCK_SCRIPT=scripts/mock_demo.txt build/hax
#
# Each `end-turn` is one stream() call. Note that an agentic chain
# (turn ends with a tool call → agent dispatches the tool → stream()
# again) consumes script turns continuously without prompting the
# user. So the user's *first* message plays everything from turn 1
# down to the first turn that ends without a tool call (turn 7 here,
# which closes the chain). The next user message picks up at turn 8
# and runs through the second chain (turns 8–9).
#
# DSL recap (see src/providers/mock.c for the full spec):
#   text <message>                    one assistant text delta
#   reasoning <message>               one chain-of-thought delta (visible
#                                     only with HAX_SHOW_REASONING=1)
#   tool <name> <json>                one tool call
#   delay <ms>                        sleep before next emission and
#                                     between auto-streamed text chunks
#   usage in=N out=M [cached=K]       set usage on the upcoming end-turn
#                                     (long-form input=/output= also work)
#   end-turn                          finalize the current turn
#   # comment / blank line            ignored

# ---------- turn 1: reasoning prelude, then a simple bash call ---------
# The reasoning block exercises the CoT render path: with HAX_SHOW_
# REASONING=1 it streams as a dim+italic span reflowed to terminal
# width; Markdown markers (the `backticks` around identifiers, the
# blank-line paragraph break) parse for structure but suppress inline
# SGR so the whole block stays one color. Without the env var the
# agent only flips the spinner label to "thinking..." and discards
# the text — matching how real providers behave.
delay 30
reasoning The user just said hello, so I should poke around to see what kind of repo this is before answering. A single `ls scripts/` is enough — the directory is flat, no need for `find` or `-R`.\n\nLet me run it and then summarize.
text Let me poke around the repo to start.
tool bash {"command":"ls scripts/"}
usage in=420 out=18
end-turn

# ---------- turn 2: slow streaming with visible spinner -----------------
# Streams 6 lines at ~0.4s each — slow enough that the live preview
# spinner repaints between commits, fast enough to keep the demo
# moving. Exercises the streaming path + spinner-on-first-byte.
text Now a slow streamer to show the spinner between chunks.
tool bash {"command":"scripts/stream_demo.py slow --delay 0.4 --count 6"}
usage in=520 out=22
end-turn

# ---------- turn 3: display head/tail elision (long mode) ---------------
# Many committed lines pass through the preview. The dim block keeps
# the first DISP_HT_HEAD_LINES, the rest go through the tail ring,
# and the elision marker appears between them. (This is the *display*
# head/tail in tool_render — the model's history truncation, exercised
# in the next turn, is a separate mechanism that stays under the cap
# here.)
text This one overruns the display head and lands content in the tail.
tool bash {"command":"scripts/stream_demo.py long --delay 0.05"}
usage in=640 out=28
end-turn

# ---------- turn 4: model-facing head+tail truncation -------------------
# ~108 KiB / 20002 lines trips bash.c's OUTPUT_CAP (50 KiB / 2000 lines),
# so the capture spills to a temp file. Two distinct things to eyeball:
#   - Display: the renderer shows head rows, its own "... (N more lines)
#     ..." elision, then the tail rows ending in the trailer — no trailing
#     truncation marker (the renderer's elision is the only signal). An
#     exit footer, if any, sits on the last row.
#   - Model history (run with HAX_TRANSCRIPT=/tmp/t.txt and grep it): the
#     body is a head slice + a mid-body marker "... [output truncated:
#     omitted N of M lines (kept first H, last T); full output saved to
#     PATH] ..." + a tail slice. The front-loaded SUMMARY line survives in
#     the head — the whole point, since tail-only truncation used to drop
#     it — and the saved path lets the model re-read the full output.
text This one is big enough that the model gets a head+tail summary with a gap marker.
tool bash {"command":"echo '=== SUMMARY (front-loaded line; must survive in the head) ==='; seq 1 20000; echo '=== TRAILER (must survive in the tail) ==='"}
usage in=700 out=30
end-turn

# ---------- turn 5: file-read preview (R_HEAD_ONLY mode) ----------------
text And here's the read tool's head-only preview.
tool read {"path":"src/render/disp.h"}
usage in=780 out=24
end-turn

# ---------- turn 6: diff preview (write tool) + compose label -----------
# Seed the file via bash first so the write below produces an actual
# unified diff. fs_write_with_diff suppresses the diff for new-file
# creates, and emits an empty payload when content is byte-identical
# to what's there — without the seed, this turn's output depends on
# whether a previous run left the file behind.
#
# The `delay 800` (set after the text so the prose still streams fast)
# also exercises the named compose spinner: the mock pauses between a
# tool call's start (name known) and its args delta, so the agent
# surfaces "[bash] composing..." for the otherwise-anonymous window
# where the model is still streaming a call's arguments. The `delay 80`
# before the second call models the small real gap between back-to-back
# tool calls: under the compose-revert threshold, so the spinner swaps
# straight to "[write] composing..." instead of flashing "working..."
# between the two (a write's `content` is the large-args case this
# stands in for).
text One write to /tmp so the diff renderer has something to color.
delay 800
tool bash {"command":"printf 'hello from a previous run\\n' > /tmp/hax_mock_demo.txt"}
delay 80
tool write {"path":"/tmp/hax_mock_demo.txt","content":"hello from the mock provider — second time around\n"}
usage in=860 out=30
end-turn

# ---------- turn 7: final assistant turn, no tool call ------------------
# Ends without a tool call, so the agent loop unwinds and control
# returns to the prompt. Type a follow-up message to start the next
# chain (turn 8).
text All done — that covered live preview, display elision, model truncation, file read, and diff coloring.
usage in=920 out=44
end-turn

# ---------- turn 8: continuation after the user types again -------------
# Picks up where turn 7 left off: a fresh chain on the second user
# message that runs a quick command, then resolves at turn 9.
text Sure, here's one more — a tiny ls just to show the continuation works.
tool bash {"command":"ls -1 src/providers"}
usage in=980 out=20
end-turn

# ---------- turn 9: slow table → "composing..." spinner ----------------
# A GFM table is the one block the renderer can't stream: column widths
# need every row, so md buffers the whole grid and emits nothing until
# the last row is in (see md_in_table / finalize_table). The intro line
# streams fast (delay 0), then `delay 250` paces the table rows in
# 16-byte chunks so the silent buffering runs well past the 1.5s idle
# threshold — long enough for the agent to raise the labeled
# "composing..." line spinner on its own row, which is erased in place
# when the finished grid lays out below the intro text.
text Here's a quick comparison of the OpenAI-family presets:
delay 250
text \n\n| Preset | Default base URL | Context probe |\n| --- | --- | --- |\n| openai | api.openai.com | none |\n| openai_compat | (your URL) | none |\n| llamacpp | 127.0.0.1:8080 | /props background |\n| openrouter | openrouter.ai/api | /endpoints background |\n
usage in=1020 out=72
end-turn
